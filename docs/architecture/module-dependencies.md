# Module Dependencies

## Purpose

Документ фиксирует допустимые зависимости между backend-модулями и правила, которые защищают модульный монолит от неявной связанности.

## Context

MVP требует быстрой разработки, но домен уже состоит из нескольких связанных областей: пользователи, группы, мероприятия, чеки, долги и платежи. Если модули будут свободно обращаться друг к другу без правил, будущая поддержка и выделение сервисов станут дорогими.

## Responsibilities

- Определить допустимые направления зависимостей.
- Объяснить роль `common` и `integrations`.
- Зафиксировать запрет на доступ к внутренним данным чужих модулей.
- Дать основу для AI-агентов и разработчиков при добавлении новой функциональности.

## Non Responsibilities

- Документ не описывает Maven configuration.
- Документ не определяет package naming.
- Документ не создает классы публичных контрактов.
- Документ не заменяет модульные документы.

## Design Decisions

Базовые правила:

- `app` может зависеть от всех модулей для сборки приложения.
- Все модули могут зависеть от `common`, если нужен общий тип или ошибка.
- Доменные модули не должны зависеть от `app`.
- Доменные модули не должны напрямую зависеть от инфраструктурных деталей внешних API.
- Внешние системы доступны через `integrations` или специализированный адаптер, согласованный с владельцем доменного модуля.
- Модуль может использовать публичный контракт другого модуля, но не его внутреннюю persistence-модель.

Рекомендуемая карта зависимостей MVP:

| Модуль | Может зависеть от | Обоснование |
| --- | --- | --- |
| `app` | все модули | Точка сборки приложения и запуск Spring Boot. |
| `auth` | `users`, `common`, `integrations` | Аутентификация через Telegram создает или обновляет пользователя. |
| `users` | `common` | Пользовательская идентичность является базовой доменной областью. |
| `groups` | `users`, `common` | Группы содержат пользователей как участников. |
| `events` | `groups`, `users`, `common` | Мероприятия создаются внутри групп и включают участников. |
| `files` | `users`, `common`, `integrations` | Файлы принадлежат пользователям или мероприятиям и могут использовать внешнее хранилище. |
| `receipts` | `events`, `users`, `files`, `common`, `integrations` | Обработка чека требует контекста мероприятия, плательщика, метаданных файла и контракта ML-сервиса. |
| `debts` | `events`, `receipts`, `users`, `common` | Расчет долгов использует участников мероприятия и выбранные позиции; часть позиций может быть создана из чека. |
| `payments` | `debts`, `users`, `common` | Платежи закрывают долги через ручное подтверждение перевода по реквизитам плательщика. |
| `notifications` | `users`, `groups`, `events`, `debts`, `payments`, `common`, `integrations` | Уведомления реагируют на доменные изменения и используют Telegram. |
| `integrations` | `common` | Адаптерам внешних систем нужны общие ошибки и технические примитивы. |
| `common` | нет | Самый нижний общий модуль без доменной зависимости. |

## Constraints

- Циклические зависимости не допускаются.
- `common` не должен зависеть от доменных модулей.
- `integrations` не должен владеть бизнес-решениями.
- Модуль не должен читать или изменять таблицы другого модуля через собственный repository.
- Общая база данных не означает общее владение всеми таблицами.
- Добавление зависимости требует обновления этого документа и документа соответствующего модуля.

## Future Evolution

- Правила зависимостей можно будет проверять build-инструментами после стабилизации структуры модулей.
- Публичные контракты можно будет вынести в API-модули, если появится необходимость выделять сервисы.
- Доменные события смогут заменить прямые вызовы для межмодульных side effects после MVP.
- Kafka topics можно вводить только после стабилизации названий событий, payload и потребителей.
