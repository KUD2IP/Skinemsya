# Error Handling

## Purpose

Документ описывает единый подход к ошибкам backend: категории, поведение API, связь с логированием и ограничения безопасности.

## Context

MVP должен быть простым в разработке, но ошибки должны быть предсказуемыми для frontend, тестировщиков и AI-агентов. Без общего подхода каждый модуль начнет возвращать ошибки в своем формате, что усложнит Mini App и интеграционные тесты.

## Responsibilities

- Определить категории ошибок.
- Зафиксировать общий формат error response на концептуальном уровне.
- Объяснить, какие ошибки логируются.
- Связать ошибки с HTTP-кодами.
- Зафиксировать единый enum-каталог ошибок для использования в коде и документации.
- Защитить API от утечки внутренних деталей.

## Non Responsibilities

- Документ не создает exception classes.
- Документ не определяет точный JSON response schema.
- Документ не описывает все возможные ошибки каждого модуля.
- Документ не заменяет security-документы.

## Design Decisions

Категории ошибок MVP:

- Validation error: некорректные входные данные.
- Authentication error: отсутствующий или недействительный JWT, некорректные Telegram init data.
- Authorization error: пользователь не имеет доступа к группе, мероприятию, чеку, долгу или платежу.
- Domain error: нарушение бизнес-правила, например попытка оплатить уже закрытый долг.
- Not found error: объект не найден или недоступен пользователю.
- Integration error: Telegram, ML-сервис, СБП или файловое хранилище недоступны либо вернули непригодный ответ.
- Infrastructure error: ошибка БД, конфигурации или runtime-среды.

HTTP-коды должны отражать поведение API:

- `400` для validation errors.
- `401` для authentication errors.
- `403` для authorization errors.
- `404` для отсутствующих или недоступных ресурсов.
- `409` для конфликтов состояния домена.
- `422` для бизнес-валидного запроса, который нельзя применить из-за доменного ограничения.
- `500` для внутренних ошибок.
- `502` или `503` для ошибок внешних систем, если backend не может выполнить сценарий.

Error response должен содержать стабильный machine-readable code, пользовательски безопасное сообщение, correlation id и, при необходимости, список validation fields. Внутренние class names, SQL, stack traces, секреты и внешние raw responses не возвращаются клиенту.

Для ошибок должен быть предусмотрен отдельный enum-каталог. Каждый элемент enum должен содержать:

- стабильный код ошибки для API и логов;
- HTTP status code;
- краткое описание для разработчиков;
- уровень критичности: `critical`, `low`, `none`.

Уровень критичности используется для логирования, мониторинга и приоритизации реакции:

- `critical`: ошибка требует внимания, потому что ломает основной сценарий, платежный flow, безопасность, целостность данных или доступность системы;
- `low`: ошибка важна для диагностики, но обычно связана с пользовательским вводом, ожидаемым бизнес-ограничением или временной проблемой без потери данных;
- `none`: ошибка не требует реакции команды, например предсказуемый отказ доступа или штатная validation error.

Enum-каталог не должен заменять текст бизнес-правил. Он нужен, чтобы backend, frontend, тесты и AI-агенты использовали одни и те же коды ошибок.

## Constraints

- Нельзя возвращать stack trace в API response.
- Нельзя смешивать domain error и infrastructure error.
- Нельзя использовать HTTP 500 для известных бизнес-ошибок.
- Нельзя создавать error code вне единого enum-каталога.
- Нельзя менять стабильный код ошибки без миграции клиентов и тестов.
- Ошибка доступа не должна раскрывать существование чужого ресурса, если это создает риск приватности.
- Integration error должен содержать достаточно информации для диагностики в логах, но не в публичном response.

## Future Evolution

- Формализация error response schema в OpenAPI.
- Каталог error codes по модулям.
- Автоматическая проверка error contracts в integration tests.
- Метрики по категориям ошибок.
- Retry policy для интеграционных ошибок после появления стабильных внешних контрактов.
