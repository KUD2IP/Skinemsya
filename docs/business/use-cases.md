# Use Cases

## Purpose

Документ описывает основные use cases MVP `skinemsya` с точки зрения пользователя и backend-системы.

## Context

MVP должен покрывать один полный путь: от Telegram-чата с bot до закрытия долга. Use cases нужны для проектирования модулей, API, тестов и onboarding нового разработчика.

## Responsibilities

- Перечислить MVP-сценарии.
- Определить участников каждого сценария.
- Зафиксировать результат каждого use case.
- Отделить обязательные сценарии от будущих улучшений.

## Non Responsibilities

- Документ не описывает REST endpoints.
- Документ не определяет UI-экраны.
- Документ не задает формат request/response.
- Документ не заменяет user flows.

## Design Decisions

| Use Case | Actor | Result |
| --- | --- | --- |
| Войти через Telegram | Пользователь | Backend проверил Telegram init data и выдал access token + refresh token. |
| Добавить bot в чат | Участник Telegram-чата | Bot отправил приветственное сообщение с кнопкой Mini App и закрепил его. |
| Открыть Mini App из чата | Пользователь | Backend создал `CHAT_LINKED`-группу с названием чата или присоединил пользователя к существующей группе. |
| Автоматически добавить участника из чата | Пользователь | Пользователь, пришедший по кнопке из чата, стал участником группы в приложении. |
| Создать группу вручную | Пользователь | Создана `STANDALONE`-группа без привязки к Telegram-чату; пользователь стал владельцем. |
| Добавить участника в группу | Владелец группы | Участник добавлен в `STANDALONE`-группу по Telegram identity. |
| Создать или отредактировать группу | Владелец | Группа создана или обновлена в рамках прав владельца. |
| Создать мероприятие | Участник группы | В группе создано мероприятие с плательщиком. |
| Отредактировать мероприятие | Участник с правами | Название, описание или плательщик изменены до финального расчета. |
| Указать реквизиты плательщика | Плательщик | В профиле пользователя доступны реквизиты для ручного перевода. |
| Просмотреть реквизиты для перевода | Должник | Должник видит сумму, получателя и реквизиты плательщика из профиля. |
| Подтвердить перевод | Должник | Должник нажал `перевел`; платеж перешел в состояние ожидания подтверждения плательщика. |
| Добавить позицию вручную | Плательщик или участник мероприятия | Создана позиция с названием, количеством и общей ценой. |
| Загрузить чек | Плательщик или участник мероприятия | Файл чека сохранен и, если доступно, передан в обработку. |
| Обработать чек | Backend + ML-сервис | Backend получил структурированный JSON, проверил его и создал или предложил позиции. |
| Отредактировать позицию | Участник с правами | Название, количество или общая цена позиции изменены до распределения. |
| Отправить на распределение | Плательщик | Позиции зафиксированы для выбора участниками. |
| Выбрать позиции | Участник мероприятия | Участник отметил позиции или доли, относящиеся к нему. |
| Добавить общую позицию | Участник с правами | Позиция распределена на всех или на выбранную группу лиц. |
| Рассчитать долги | Backend | Созданы долги между участниками и плательщиком. |
| Подтвердить получение | Плательщик | Платеж подтвержден, долг закрыт. |
| Посмотреть общую статистику | Пользователь | Пользователь видит, кто сколько должен ему и кому должен он. |
| Отредактировать профиль | Пользователь | Обновлены реквизиты, телефон или настройки уведомлений. |
| Выйти из группы | Участник группы | Пользователь покинул группу, если это не нарушает активные расчеты. |
| Удалить мероприятие | Пользователь с правами | Мероприятие удалено или помечено удаленным по правилам доступа. |
| Удалить группу | Владелец группы | Группа удалена или помечена удаленной по правилам доступа. |

## Constraints

- Use cases MVP должны покрывать основной продуктовый путь из Telegram-чата и альтернативный путь ручного создания группы.
- Use case не должен требовать банковской интеграции.
- Use case не должен требовать Kafka или выделенного сервиса.
- Каждый use case должен быть тестируемым через backend-сценарий.
- Ошибка или отказ должны быть описаны как альтернативный сценарий в user flows или тестовых данных.

## Future Evolution

- Продвинутые приглашения через Telegram-ссылки вне группового чата.
- Несколько чеков на одно мероприятие.
- Частичные платежи.
- Автоматическое подтверждение СБП.
- Расширенная история и аналитика расходов.
