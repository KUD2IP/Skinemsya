# MVP Scope

## Purpose

Документ фиксирует границы MVP `skinemsya`: что должно быть реализовано в первой версии и что сознательно исключено.

## Context

Цель MVP — быстро вывести рабочий продукт на рынок и проверить основной сценарий разделения счета. Для этого проект должен избегать enterprise-решений, микросервисов, Kafka и сложной платежной интеграции до подтверждения продуктовой ценности.

## Responsibilities

- Зафиксировать обязательный объем MVP.
- Зафиксировать исключения из MVP.
- Дать критерии готовности первой версии.
- Защитить проект от преждевременного усложнения.

## Non Responsibilities

- Документ не является roadmap по датам.
- Документ не описывает backlog задач.
- Документ не создает API или миграции.
- Документ не заменяет post-MVP features.

## Design Decisions

В MVP входит:

- Telegram authentication через init data.
- Access token и refresh token.
- Добавление bot в Telegram-чат с приветственным закрепленным сообщением и кнопкой Mini App.
- Создание группы приложения из Telegram-чата (`CHAT_LINKED`).
- Ручное создание группы в Mini App без привязки к чату (`STANDALONE`).
- Присоединение пользователя к группе при входе из Telegram-чата.
- Автоматическое добавление участников после их входа в Mini App из группового чата.
- Ручное добавление участников в standalone-группу владельцем.
- Базовое редактирование группы.
- Создание мероприятия.
- Базовое редактирование мероприятия.
- Выбор плательщика.
- Возможность сменить плательщика до финального расчета.
- Указание реквизитов плательщика для ручного перевода.
- Редактирование профиля пользователя: реквизиты, телефон, настройки уведомлений.
- Ручное добавление позиций с названием, количеством и общей ценой.
- Редактирование позиций до отправки на распределение.
- Общие позиции с делением на всех или выбранную группу лиц.
- Загрузка чека.
- Интеграция с Python ML-сервисом для получения структурированного JSON.
- Backend-валидация и нормализация чека.
- Создание позиций из чека как дополнительный способ заполнения списка.
- Отправка позиций на распределение.
- Выбор позиций участниками.
- Расчет долгов.
- Отображение статусов долга: `не оплачено`, `ожидает подтверждения`, `оплачено`.
- Ручное подтверждение перевода должником.
- Ручное подтверждение получения плательщиком.
- Закрытие долга после двухстороннего подтверждения.
- Главный экран со статистикой: кто сколько должен пользователю и кому должен пользователь.
- Выход из группы с ограничениями для активных расчетов.
- Удаление мероприятия пользователем с правами.
- Удаление группы только владельцем.
- Базовые ошибки, логи и интеграционные тесты.

В MVP не входит:

- Микросервисы.
- Kafka.
- Полноценная СБП-интеграция.
- Автоматическое банковское подтверждение платежа.
- Частичные платежи.
- Оплата разных мест разными людьми.
- Сложные способы деления позиций сверх деления на всех или выбранную группу лиц.
- Продвинутая аналитика.
- Полноценный audit/event sourcing.
- Сложная ролевая модель: кастомные права, администраторы, делегирование.

## Constraints

- Новая функция входит в MVP только если она нужна для полного основного сценария.
- Инфраструктура входит в MVP только если без нее нельзя надежно запустить продукт.
- Документированные исключения не должны реализовываться случайно через побочные задачи.
- Все решения должны оцениваться по скорости вывода продукта на рынок.
- MVP должен оставаться разворачиваемым как один backend-сервис.

## Future Evolution

- Post-MVP roadmap должен добавлять функции только после запуска базового сценария.
- СБП может стать отдельным этапом после ручных платежей.
- Event Driven Architecture может появиться после стабилизации доменных событий.
- Выделение сервисов возможно только после доказанной нагрузки или независимого жизненного цикла.
