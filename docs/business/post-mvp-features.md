# Post-MVP Features

## Purpose

Документ фиксирует функции и архитектурные направления, которые не входят в MVP, но могут развивать `skinemsya` после проверки основного сценария.

## Context

Проект должен иметь возможность масштабироваться, но не должен внедрять сложность заранее. Post-MVP список нужен, чтобы сохранять идеи, не превращая их в обязательства первого релиза.

## Responsibilities

- Зафиксировать возможные улучшения после MVP.
- Отделить будущие функции от текущего объема.
- Дать основу для roadmap после запуска.
- Обозначить архитектурные направления развития.

## Non Responsibilities

- Документ не обещает реализацию всех перечисленных функций.
- Документ не задает приоритеты по датам.
- Документ не расширяет MVP автоматически.
- Документ не заменяет scaling roadmap.

## Design Decisions

Функциональные направления:

- Полноценная СБП-интеграция.
- Частичные платежи.
- Оплата разных мест разными людьми.
- Несколько чеков в мероприятии.
- Сложное деление: проценты, веса, ручные суммы.
- Комментарии к спорным платежам.
- Расширенная история групп и мероприятий.
- Продвинутая аналитика расходов.
- Приглашения через Telegram вне автоматического добавления из группового чата.
- Расширенная ролевая модель.
- Уведомления через Telegram Bot API.

Архитектурные направления:

- Доменные события внутри монолита.
- Асинхронная обработка чеков.
- Kafka при появлении независимых потребителей.
- Выделение `receipts`, `payments` или `notifications` в отдельные сервисы.
- Расширенный audit log.
- Централизованные метрики и tracing.

## Constraints

- Post-MVP функция не должна попадать в MVP без явного решения.
- Новая инфраструктура должна иметь измеримую причину.
- СБП не должен ломать ручной платежный сценарий: он может остаться резервным.
- Выделение сервиса не должно происходить только из-за теоретической масштабируемости.
- Будущие события должны сначала стабилизироваться в доменной модели.

## Future Evolution

Документ должен обновляться после пользовательских интервью, тестирования MVP и появления данных о реальной нагрузке. После запуска часть пунктов должна перейти в roadmap с приоритетами, критериями готовности и зависимостями.
