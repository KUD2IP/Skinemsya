# Business Rules

## Purpose

Документ фиксирует бизнес-правила MVP `skinemsya`: создание групп, обработку мероприятий и чеков, выбор позиций, расчет долгов и закрытие оплаты.

## Context

Backend должен одинаково применять правила во всех сценариях: через API, тесты, будущие уведомления и возможные интеграции. Если правила не зафиксированы явно, расчет долгов и статусы оплаты станут зависеть от конкретной реализации.

## Responsibilities

- Зафиксировать правила основного MVP-сценария.
- Определить условия расчета долгов.
- Описать правила двухстороннего подтверждения оплаты.
- Дать основу для тест-кейсов.

## Non Responsibilities

- Документ не определяет алгоритм на уровне кода.
- Документ не создает enum-значения.
- Документ не описывает структуру таблиц.
- Документ не описывает UI.

## Design Decisions

Чат, группы и участники:

- Группа может быть двух типов: `CHAT_LINKED` (связана с Telegram-чатом) или `STANDALONE` (создана вручную без чата).
- Групповой Telegram-чат является основным входом для `CHAT_LINKED`-групп.
- Bot должен быть добавлен в Telegram-чат с правами, достаточными для отправки и закрепления приветственного сообщения.
- При входе в Mini App по кнопке из чата backend должен создать `CHAT_LINKED`-группу с названием чата или присоединить пользователя к уже созданной группе.
- Пользователь может создать `STANDALONE`-группу напрямую в Mini App, указав название.
- Владелец `STANDALONE`-группы добавляет участников вручную по Telegram identity.
- Если пользователь зашел в мероприятие из чата позже остальных, он должен получить доступ к группе и мероприятию после успешной Telegram-аутентификации.
- Пользователь может работать только с группами, где он является участником.
- Мероприятие может быть создано только внутри существующей группы.
- Участник мероприятия должен быть участником группы.
- Группу можно редактировать, если у пользователя есть права владельца.
- Удаление группы должно быть доступно только владельцу.

Мероприятие:

- Мероприятие создается внутри группы.
- Мероприятие должно иметь название.
- Описание мероприятия является опциональным.
- Плательщик выбирается из участников группы.
- Плательщика можно сменить до финального расчета или по явно описанному правилу пересчета.
- Мероприятие можно редактировать до момента, когда расчеты считаются финальными.
- Удаление мероприятия должно быть ограничено владельцем мероприятия или владельцем группы.

Позиции и чек:

- Чек должен быть связан с мероприятием.
- Позиции можно добавлять вручную без чека.
- Ручная позиция должна содержать название, количество и общую цену с учетом количества.
- Загрузка чека является дополнительным способом заполнения позиций.
- Backend принимает структурированный JSON после ML-обработки, если позиция создается из чека.
- Backend обязан проверить суммы, количества и обязательные поля до создания доменной модели.
- Позиции можно редактировать до отправки на распределение.
- Позиция может быть выбрана одним или несколькими участниками.
- Общую позицию можно сразу разделить на всех участников мероприятия или на выбранную группу лиц.
- Если позиция выбрана несколькими участниками, стоимость делится между ними по правилам MVP.
- После окончательного формирования позиций плательщик отправляет мероприятие на распределение.

Расчет долгов:

- Расчет выполняется после отправки позиций на распределение и завершения выбора участниками.
- Базовый получатель долга в MVP — плательщик мероприятия.
- Долг создается для каждого участника, который должен вернуть часть расходов.
- Плательщик не должен иметь долг перед самим собой.
- Долг должен иметь понятный статус: `не оплачено`, `ожидает подтверждения`, `оплачено`.
- Денежные значения должны рассчитываться в минимальных денежных единицах или через точный decimal-подход, чтобы избежать ошибок floating point.
- Округление должно быть детерминированным и одинаковым для backend и тестов.

Оплата:

- Реквизиты для перевода хранятся в профиле пользователя (плательщика).
- Плательщик указывает или обновляет реквизиты в профиле до оплаты долга.
- Должник нажимает `перевести` и видит сумму, получателя и реквизиты из профиля плательщика.
- После перевода вне системы должник нажимает `перевел`.
- Долг не закрывается после действия должника.
- Долг закрывается только после действия плательщика `получил`.
- Если плательщик не подтверждает получение, платеж остается в состоянии ожидания или спора.

Маппинг статусов долга и платежа:

| Статус долга | Статус платежа | Условие |
| --- | --- | --- |
| `не оплачено` | нет платежа или `created` | Долг рассчитан, оплата не начата |
| `ожидает подтверждения` | `debtor_confirmed` | Должник нажал `перевел` |
| `оплачено` | `payer_confirmed` | Плательщик нажал `получил` |

## Business Invariants

Эти правила являются обязательными для любой реализации, тестов и будущих API:

### Пользователи и группы

- Один Telegram user id соответствует одному backend user.
- Пользователь становится участником `CHAT_LINKED`-группы только после входа в Mini App и успешной проверки Telegram init data.
- Наличие пользователя в Telegram-чате само по себе не создает backend-доступ, пока пользователь не прошел auth.
- `STANDALONE`-группа создается только аутентифицированным пользователем; создатель становится `owner`.
- В `STANDALONE`-группе ручное добавление участника требует, чтобы пользователь уже имел Telegram identity в системе или был добавлен через явно описанный invite flow.
- В MVP существует минимальная роль `owner`; все остальные участники имеют роль `member`.

### Мероприятия

- Мероприятие всегда принадлежит одной группе.
- Плательщик должен быть участником группы на момент назначения.
- Если плательщик меняется до финального расчета, долги еще не должны быть рассчитаны или должны быть явно пересчитаны.
- После расчета долгов редактирование позиций и плательщика запрещено, если не реализован отдельный flow перерасчета.

### Позиции

- Ручные позиции и позиции из чека должны приводиться к единой доменной модели `Position`.
- Пользовательский выбор позиции должен быть связан с участником мероприятия, а не просто с произвольным user id.
- Общая позиция делится только между теми пользователями, которые являются участниками мероприятия.
- Позиция с нулевой или отрицательной стоимостью не входит в расчет долга.
- Количество позиции влияет на отображение и проверку чека, но расчет долга в MVP использует итоговую стоимость позиции.

### Долги

- Долг создается только по финальному набору позиций и выборов.
- Для одного мероприятия можно пересчитать долги только до начала платежей. После подтверждения хотя бы одного платежа перерасчет требует отдельного future flow.
- Сумма всех долгов должна быть равна сумме долей, которые должны вернуться плательщику, с учетом deterministic rounding.
- Остаток округления должен распределяться детерминированно, чтобы повторный расчет давал тот же результат.

### Платежи

- Один долг имеет не более одной активной платежной операции.
- `перевести` не меняет финансовый статус; это только просмотр реквизитов.
- `перевел` меняет payment status, но не закрывает debt.
- `получил` закрывает payment и debt.
- Спорный платеж не должен автоматически закрывать debt.

## Edge Cases

| Ситуация | Ожидаемое поведение |
| --- | --- |
| Пользователь открыл Mini App вне чата | Можно создать `STANDALONE`-группу |
| Пользователь из чата впервые открыл Mini App | Создать user, затем добавить в `CHAT_LINKED`-группу |
| Bot не может закрепить сообщение | Группа не создается автоматически; ошибка относится к Telegram integration |
| Плательщик не указал реквизиты | Должник не может выполнить `перевести`; показывается ошибка действия |
| ML-сервис недоступен | Чек FAILED/PROCESSING timeout, ручные позиции остаются доступными |
| Чек распознан с низким confidence | Позиции требуют явного подтверждения или пометки low confidence |
| Участник вышел из группы с активным долгом | Выход блокируется или требует закрытия активных расчетов |
| Плательщик нажал `получил` до `перевел` | Ошибка состояния payment |
| Должник повторно нажал `перевел` | Idempotent response или ошибка duplicate action |
| Пользователь пытается выбрать позицию чужого мероприятия | Authorization error |

## Testable Acceptance Rules

Минимальный набор бизнес-тестов:

- `CHAT_LINKED` group создается по chat id и повторный вход не создает дубль.
- `STANDALONE` group создается без `telegramChatId`, owner добавляется в members.
- Участник не может создать event в группе, где он не member.
- Плательщик не может быть пользователем вне группы.
- Позиции можно добавить без чека.
- Receipt JSON с пустым `items` не создает positions.
- Расчет долгов не создает долг плательщика перед собой.
- `перевел` переводит долг в `ожидает подтверждения`.
- `получил` переводит долг в `оплачено`.
- Нельзя удалить группу с активными незакрытыми долгами.

## Constraints

- Нельзя рассчитывать долги по непроверенному чеку, если позиции были созданы из результата распознавания.
- Нельзя рассчитывать долги по позициям, которые еще не отправлены на распределение.
- Нельзя закрывать долг без подтверждения плательщика.
- Нельзя использовать СБП как обязательную MVP-зависимость.
- Нельзя хранить лишние платежные данные.
- Нельзя скрывать спорные или незавершенные платежи от участников сценария.
- Нельзя использовать Telegram username как стабильный идентификатор.

## Future Evolution

- Частичные платежи.
- Оплата разных мест разными людьми.
- Сложные способы деления: проценты, веса, ручные суммы.
- Несколько плательщиков в одном мероприятии.
- Несколько чеков в одном мероприятии.
- Автоматическое подтверждение через СБП.
- Доменные события для перерасчетов и уведомлений.
