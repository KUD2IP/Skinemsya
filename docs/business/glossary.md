# Glossary

## Purpose

Документ фиксирует единые термины проекта `skinemsya`, чтобы разработчики, тестировщики, архитекторы, DevOps-инженеры и AI-агенты использовали один словарь.

## Context

Проект описывает разделение счета между участниками группы. В домене есть бытовые термины пользователей и технические термины backend. Без общего словаря документы и код быстро начнут использовать разные названия для одного и того же понятия.

## Responsibilities

- Определить основные бизнес-термины.
- Связать пользовательские понятия с backend-понятиями.
- Уменьшить неоднозначность в требованиях, тестах и документации.
- Зафиксировать термины MVP и отделить их от post-MVP идей.

## Non Responsibilities

- Документ не описывает таблицы БД.
- Документ не задает имена классов или DTO.
- Документ не описывает REST API.
- Документ не заменяет бизнес-правила.

## Design Decisions

| Термин | Значение |
| --- | --- |
| Пользователь | Человек, открывающий `skinemsya` через Telegram Mini App. |
| Telegram identity | Внешняя идентичность пользователя из Telegram, включая Telegram user id. |
| Backend user id | Внутренний идентификатор пользователя в backend. Используется в доменных связях вместо Telegram user id. |
| Чат | Групповой чат в Telegram с участниками мероприятия. В MVP чат является основным источником создания или присоединения к группе в приложении. |
| Бот | Telegram bot, добавляемый в чат с правами администратора. Отправляет и закрепляет приветственное сообщение с кнопкой открытия Mini App. |
| Группа | Объединение пользователей внутри приложения. Тип `CHAT_LINKED` — связана с Telegram-чатом; тип `STANDALONE` — создана вручную. Внутри группы создаются мероприятия. |
| Chat-linked группа | Группа, автоматически созданная или присоединённая при входе из Telegram-чата. Имеет обязательную привязку `telegramChatId`. |
| Standalone группа | Группа, созданная пользователем вручную в Mini App без привязки к чату. Участники добавляются владельцем. |
| Владелец группы | Минимальная MVP-роль: создатель группы, может редактировать и удалять группу, добавлять участников в standalone-группу. |
| Участник группы | Пользователь, добавленный в группу. |
| Мероприятие | Конкретный случай совместных расходов внутри группы. Backend-сущность: `Event`. В UI называется **сбор**. |
| Сбор | Пользовательское название мероприятия (`Event`) в Mini App и чат-боте. |
| Плательщик | Участник, который оплатил общий чек вне системы и ожидает возврата долей. |
| Чек | Изображение или структурированный результат обработки чека, который может использоваться для автоматического создания позиций. На MVP позиции можно добавлять и без чека. |
| Позиция | Строка расхода в мероприятии: товар или услуга с названием, количеством и общей ценой с учетом количества. Может быть создана вручную или получена из обработанного чека. |
| Общая позиция | Позиция, которая делится на всех участников мероприятия или на выбранную группу участников. |
| Выбор позиции | Действие пользователя, которым он отмечает, какие позиции относятся к нему. |
| Доля | Часть стоимости позиции, относящаяся к конкретному участнику. |
| Долг | Сумма, которую один участник должен заплатить плательщику или другому получателю по результатам расчета. |
| Статус долга | Состояние долга. Backend: `не оплачено`, `ожидает подтверждения`, `оплачено`. UI: **Не скинул**, **Ждёт проверки**, **Подтверждено**. |
| Реквизиты плательщика | Данные для ручного перевода денег. Хранятся в профиле пользователя. |
| Просмотр реквизитов | Действие должника перед переводом. Backend: `перевести`. UI: **Скинуть**. |
| Подтверждение должника | Должник сообщает, что отправил деньги вне системы и прикрепляет скрин перевода. Backend: `перевел` → `debtor_confirmed`. UI: **Отправил**. |
| Скрин перевода | Изображение подтверждения банковского перевода, прикрепляемое должником при действии «Отправил». Хранится в `files`, связан с `Payment`. |
| Профиль пользователя | Настройки пользователя: реквизиты для переводов, телефон и настройки уведомлений. |
| Платеж | Запись backend о попытке закрыть конкретный долг. |
| Подтверждение плательщика | Плательщик подтверждает получение денег. Backend: `получил` → `payer_confirmed`. UI: **Подтвердить** (индивидуально) или **Всё на месте** (массово). |
| Оспаривание перевода | Плательщик сообщает, что перевод не поступил. Backend: `disputed`. UI: **Не пришло**. |
| Закрытый долг | Долг, по которому выполнено подтверждение плательщика. |
| Запуск сбора | Отправка позиций на выбор участникам. Backend: `sendToDistribution` → статус `DISTRIBUTION`. UI: **Запустить сбор**. |
| Python ML-сервис | Внешний сервис, выполняющий OCR/ML-распознавание изображения чека и возвращающий структурированный JSON. |
| Структурированный JSON чека | Результат ML-сервиса, который backend валидирует и превращает в доменные модели. |
| MVP | Минимальная версия продукта, покрывающая основной сценарий без микросервисов, Kafka и банковской интеграции. |

### UI-термины и соответствие backend

| UI / чат-бот | Backend / docs | Примечание |
| --- | --- | --- |
| Сбор | `Event` / мероприятие | Только пользовательский язык |
| Готовится | `DRAFT` | Статус сбора |
| Выбирают | `DISTRIBUTION` | Статус сбора |
| Закрыт | `COMPLETED` | Статус сбора |
| Запустить сбор | `sendToDistribution` | Действие плательщика |
| Позиция | `Position` | При редактировании списка |
| Блюдо | `Position` | При выборе участником |
| Скинуть | `перевести` | Открыть реквизиты |
| Отправил | `перевел` / `debtor_confirmed` | С прикреплением скрина |
| Не скинул | `не оплачено` / `UNPAID` | Статус долга |
| Ждёт проверки | `ожидает подтверждения` / `debtor_confirmed` | Скрин приложен |
| Подтверждено | `оплачено` / `payer_confirmed` | Долг закрыт |
| Всё на месте | bulk `получил` | Массовое подтверждение плательщика |
| Не пришло | `disputed` | Оспаривание перевода |
| На всех | общая позиция | Деление на всех участников |

## Constraints

- Термины должны использоваться одинаково во всех документах.
- `Плательщик` не означает банк или платежного провайдера.
- `Платеж` в MVP не означает автоматическую банковскую транзакцию.
- `Чек` в backend не является сырым изображением; изображение относится к файловому контуру.
- `Позиция` не обязана происходить из чека: ручное добавление является базовым сценарием.
- `СБП` не является частью MVP-платежного сценария, а относится к будущему развитию.

## Future Evolution

- Расширить термины для ролей группы: владелец, администратор, участник.
- Добавить термины для частичных платежей.
- Добавить термины для СБП-интеграции после ее проектирования.
- Добавить термины для доменных событий при переходе к Event Driven Architecture.
