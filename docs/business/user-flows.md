# User Flows

## Purpose

Документ описывает пользовательские сценарии MVP `skinemsya` от входа в Mini App до закрытия долга.

## Context

User flows нужны, чтобы backend-разработчики понимали последовательность действий пользователя, а тестировщики могли строить end-to-end сценарии без чтения исходного кода.

## Responsibilities

- Описать основной успешный пользовательский путь.
- Описать ключевые альтернативные сценарии.
- Связать пользовательские действия с бизнес-состояниями.
- Показать, где возникают ошибки и ожидание действий других участников.

## Non Responsibilities

- Документ не проектирует UI-компоненты.
- Документ не задает тексты кнопок как окончательные product copy.
- Документ не описывает REST endpoints.
- Документ не заменяет use cases.

## Design Decisions

Основной поток MVP описан детально в `docs/product/ux-checklist.md`. Ниже — краткая последовательность с backend-состояниями. В UI используются термины из glossary (сбор, скинуть, отправил, всё на месте).

Основной поток MVP:

1. Пользователь заходит в Telegram.
2. Пользователь открывает существующий групповой чат или создает новый чат с участниками.
3. В чат добавляется bot с правами администратора.
4. Bot отправляет приветственное сообщение с кнопкой открытия Mini App и закрепляет сообщение.
5. Пользователь открывает Mini App по кнопке из чата.
6. Backend проверяет Telegram init data и выдает access token + refresh token.
7. Backend определяет Telegram-чат и создает группу приложения с названием чата или присоединяет пользователя к уже созданной группе.
8. Пользователь создает сбор (`Event`) внутри группы. Создатель автоматически выбран плательщиком.
9. Пользователь указывает название сбора (автозаполнение из чата) и, при необходимости, заметку.
10. Плательщика можно сменить до запуска сбора.
11. Плательщик добавляет позиции вручную или загружает чек для автоматического парсинга (ML).
12. Пользователь редактирует позиции, помечает общие позиции («На всех»), делит чаевые.
13. Плательщик нажимает **Запустить сбор** (`sendToDistribution` → `DISTRIBUTION`).
14. Bot отправляет в групповой чат уведомление с кнопкой **Скинуть** (`startapp=event_{eventId}`).
15. Участники выбирают блюда (позиции) с ограничением по количеству.
16. Backend рассчитывает долги.
17. Должник нажимает **Скинуть**, видит реквизиты, переводит деньги вне системы.
18. Должник прикрепляет скрин перевода и нажимает **Отправил** (`debtor_confirmed`).
19. Плательщик видит статусы: Не скинул / Ждёт проверки / Подтверждено.
20. Плательщик нажимает **Всё на месте** (bulk confirm) или **Не пришло** по конкретному участнику.
21. Backend закрывает долги (`payer_confirmed`); сбор переходит в `COMPLETED` когда все подтверждены.

Альтернативный поток: ручное создание группы:

1. Пользователь открывает Mini App напрямую (без входа из чата).
2. Backend проверяет Telegram init data и выдает access token + refresh token.
3. Пользователь создает `STANDALONE`-группу, указывая название.
4. Пользователь добавляет участников по Telegram identity.
5. Далее сценарий совпадает с шагами 8–21 основного потока.

Альтернативные сценарии:

- ML-сервис вернул неполный JSON: чек требует ручного дополнения на том же экране.
- Участники еще не зашли в Mini App: отображаются как «Ещё не зашёл»; напоминание через бота.
- Позиции добавляются без чека: сбор может полностью пройти сценарий ручного добавления.
- Участник не выбрал блюда: сбор остается неготовым к расчету.
- Должник нажал «Отправил», плательщик нажал «Не пришло»: платеж в `disputed`; бот уведомляет должника.
- Плательщик не указал реквизиты: запуск сбора заблокирован с CTA в профиль.
- Пользователь пытается открыть чужую группу: backend возвращает authorization error.
- Владелец редактирует группу, мероприятие или позиции до момента, когда изменения запрещены бизнес-правилами.
- Владелец удаляет группу, а мероприятие удаляется пользователем с достаточными правами.

## Constraints

- Основной поток не должен требовать интеграции с банком.
- Расчет долгов не должен запускаться до отправки позиций на распределение и выбора участниками.
- Закрытие долга не должно происходить без действия плательщика (массового или индивидуального).
- Скрин перевода обязателен для подтверждения должника на MVP.
- Сценарий должен оставаться коротким: MVP не должен добавлять роли, согласования и сложные настройки без необходимости.
- Backend-состояния должны быть понятны frontend без скрытых переходов.

## Future Evolution

- Более гибкие уведомления участникам на каждом шаге.
- Автоматический переход к СБП-оплате.
- Повторное открытие мероприятия для исправления.
- Спорные платежи с комментариями.
- Архив групп и расширенная история расходов.
