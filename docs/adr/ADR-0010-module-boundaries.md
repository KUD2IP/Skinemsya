# ADR-0010: Module Boundaries

## Status

Accepted

## Purpose

Зафиксировать правила границ модулей внутри модульного монолита `skinemsya`.

## Context

Модульный монолит дает скорость MVP только если границы модулей понятны и соблюдаются. Без правил проект быстро станет обычным монолитом с неявными зависимостями, где изменения в платежах ломают чеки, а модуль групп напрямую меняет таблицы пользователей.

## Responsibilities

- Определить принцип владения доменными объектами.
- Запретить неявный доступ к внутренним данным модулей.
- Зафиксировать публичные контракты как способ межмодульного взаимодействия.
- Поддержать будущую возможность выделения сервисов.

## Non Responsibilities

- ADR не описывает конкретные package names.
- ADR не создает API interfaces.
- ADR не определяет все доменные события.
- ADR не заменяет `module-dependencies.md`.

## Design Decisions

Каждый модуль владеет своей доменной областью, правилами, persistence-объектами и публичными контрактами.

Другой модуль может использовать только публичный контракт владельца. Прямой доступ к internal classes, repositories, entities или таблицам другого модуля запрещен как архитектурное нарушение.

`common` используется только для действительно общих технических или базовых типов. `integrations` содержит адаптеры внешних систем, но не принимает бизнес-решения.

Общая PostgreSQL database на MVP не отменяет владение таблицами. Таблица принадлежит модулю, даже если физически находится в общей базе.

## Constraints

- Циклические зависимости запрещены.
- Межмодульная зависимость должна быть явной.
- Новый публичный контракт должен быть описан в документации модуля.
- Модуль не должен менять чужую таблицу через собственный repository.
- Бизнес-правила не должны уходить в `common` или `integrations`.

## Future Evolution

Публичные контракты модулей могут стать API выделенных сервисов. Доменные события могут стать событиями Kafka, если появятся независимые потребители. До этого границы нужны не для усложнения, а для управляемой разработки MVP.
