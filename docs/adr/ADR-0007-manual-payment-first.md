# ADR-0007: Manual Payment First

## Status

Accepted

## Purpose

Зафиксировать ручной платежный сценарий на MVP: перевод по реквизитам плательщика и двухстороннее подтверждение оплаты.

## Context

Основная ценность MVP состоит в том, чтобы быстро помочь группе разделить чек, понять кто кому должен и закрыть долг. Полноценная СБП-интеграция потребует дополнительных внешних контрактов, юридических и технических проверок, обработки callback-уведомлений и более строгих требований к платежным данным.

Для первого релиза достаточно поддержать простой сценарий: плательщик указывает реквизиты, должник переводит деньги вне системы, затем должник отмечает `перевел`, а плательщик отмечает `получил`.

## Responsibilities

- Зафиксировать платежный MVP без банковской интеграции.
- Определить двухстороннее подтверждение как условие закрытия долга.
- Сохранить возможность будущего перехода к СБП.
- Ограничить хранение платежных данных.

## Non Responsibilities

- ADR не описывает банковское API.
- ADR не создает платежные endpoints.
- ADR не определяет конкретный формат реквизитов.
- ADR не гарантирует автоматическую проверку факта перевода.

## Design Decisions

MVP использует ручной платежный сценарий:

1. Плательщик указывает реквизиты в профиле пользователя.
2. Должник нажимает `перевести` и видит сумму, получателя и реквизиты.
3. Должник выполняет перевод вне системы.
4. Должник нажимает `перевел`.
5. Плательщик нажимает `получил`.
6. Только после подтверждения плательщика долг закрывается.

Такой подход выбран потому, что он быстрее реализуется, проще тестируется и не требует ранней интеграции с банками. Он сохраняет продуктовую ценность MVP: расчет и согласование долгов.

## Constraints

- Долг нельзя закрыть только по подтверждению должника.
- Реквизиты должны храниться только в объеме, нужном для перевода между участниками.
- Система не должна утверждать, что перевод реально совершен, пока плательщик не подтвердил получение.
- Спорные ситуации должны оставаться видимыми в статусах.
- СБП не является MVP-зависимостью.

## Future Evolution

СБП может стать основным платежным сценарием после MVP. Для перехода нужно определить контракт интеграции, статусы платежей, подтверждение оплаты, обработку ошибок, требования к безопасности и возможные webhooks.

Ручной сценарий может остаться резервным вариантом, если СБП недоступен или пользователь предпочитает другой способ перевода.
