# ADR-0008: Receipt Processing Boundaries

## Status

Accepted

## Purpose

Зафиксировать границы ответственности backend при обработке чеков.

## Context

Позиции мероприятия можно добавлять вручную. Пользователь также может загрузить чек, чтобы автоматически заполнить позиции, но backend `skinemsya` не должен выполнять OCR. На MVP распознавание изображения делает отдельный Python ML-сервис. Backend получает структурированный JSON и отвечает за его проверку, нормализацию и перевод в позиции мероприятия.

## Responsibilities

- Зафиксировать, что backend не занимается OCR.
- Определить, что backend валидирует структурированный JSON.
- Зафиксировать, что ручное добавление позиций не зависит от обработки чека.
- Разделить ML-распознавание и бизнес-парсинг.
- Поддержать будущую асинхронную обработку чеков.

## Non Responsibilities

- ADR не описывает ML-модель.
- ADR не определяет полный JSON contract.
- ADR не создает receipt entities.
- ADR не описывает UI ручного исправления чека.

## Design Decisions

Backend принимает только структурированный результат распознавания, если позиции создаются из чека. Он не использует сырой OCR-текст как источник бизнес-правды. Ручные позиции создаются напрямую в домене мероприятия и не проходят через ML-контур.

Модуль `receipts` отвечает за:

- проверку обязательных полей;
- проверку сумм и количества;
- нормализацию денежных значений;
- создание доменной модели чека и позиций мероприятия;
- фиксацию состояния, если данные требуют исправления.

ML-сервис отвечает за извлечение данных из изображения и оценку уверенности распознавания. Backend не принимает ML-результат без доменной валидации.

## Constraints

- Нельзя создавать долги на основе непроверенного JSON.
- Нельзя блокировать ручное добавление позиций из-за недоступности ML-сервиса.
- Нельзя считать ML confidence бизнес-решением без backend-валидации.
- Нельзя переносить расчет долгов в ML-сервис.
- Нельзя добавлять Kafka для обработки чеков на MVP.
- Ошибки распознавания должны быть представлены понятным доменным состоянием.

## Future Evolution

- Асинхронная обработка чеков.
- Повторное распознавание.
- Ручное исправление позиций.
- Хранение confidence по строкам чека.
- Доменные события `ReceiptProcessed`, `ReceiptFailed`, `ReceiptConfirmed`.
