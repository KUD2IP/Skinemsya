# ADR-0009: Python ML Service

## Status

Accepted

## Purpose

Зафиксировать использование отдельного Python ML-сервиса для OCR и распознавания чеков.

## Context

OCR и ML-распознавание чеков требуют другого технологического стека, чем Java backend. Python имеет более подходящую экосистему для ML/OCR-инструментов. Попытка встроить OCR в backend усложнит Java-приложение и смешает бизнес-логику с экспериментальным ML-контуром.

## Responsibilities

- Зафиксировать отдельный ML-контур.
- Определить границу между Java backend и Python-сервисом.
- Сохранить простоту backend MVP.
- Поддержать развитие ML независимо от доменной логики.

## Non Responsibilities

- ADR не выбирает OCR-библиотеку.
- ADR не описывает обучение модели.
- ADR не определяет инфраструктуру ML-сервиса.
- ADR не создает контракт JSON.

## Design Decisions

Python ML-сервис отвечает за распознавание изображения чека и возврат структурированного JSON. Java backend отвечает за бизнес-валидацию и доменную обработку результата.

На MVP интеграция может быть синхронной, если время ответа укладывается в приемлемый пользовательский сценарий. Если распознавание станет долгим или нестабильным, обработка должна перейти в асинхронный режим.

## Constraints

- ML-сервис не должен иметь права изменять доменные данные backend напрямую.
- ML-сервис не должен рассчитывать долги.
- Backend не должен доверять ML-результату без проверки.
- Секреты интеграции не должны храниться в коде.
- Ошибки ML-сервиса должны быть нормализованы backend.

## Future Evolution

- Асинхронный processing через очередь.
- Несколько ML-пайплайнов для разных типов чеков.
- Метрики качества распознавания.
- Версионирование JSON-контракта.
- Выделенный lifecycle deployment для ML-сервиса.
