# ADR-0001: Modular Monolith

## Status

Accepted

## Purpose

Зафиксировать выбор модульного монолита как базовой архитектуры backend `skinemsya` на этапе MVP.

## Context

`skinemsya` должен быстро вывести на рынок Telegram Mini App для разделения счетов. Основной сценарий включает группы, мероприятия, чеки, выбор позиций, расчет долгов и подтверждение оплат. Эти области связаны между собой транзакционно и пока не требуют независимых команд, отдельных релизных циклов или горизонтального масштабирования по сервисам.

Микросервисы и Kafka на старте увеличат стоимость разработки: потребуется распределенная трассировка, orchestration, межсервисные контракты, сетевые ошибки, отдельные базы или строгие схемы владения данными. Для MVP эта сложность не дает пропорциональной бизнес-ценности.

## Responsibilities

- Зафиксировать архитектурный стиль backend.
- Объяснить отказ от микросервисов на MVP.
- Сохранить модульные границы внутри единого приложения.
- Поддержать будущий переход к выделенным сервисам без раннего усложнения.

## Non Responsibilities

- ADR не определяет физическую Maven-структуру.
- ADR не описывает конкретные классы, REST API или миграции.
- ADR не запрещает выделение сервисов после MVP.
- ADR не заменяет документ `backend-architecture.md`.

## Design Decisions

Backend реализуется как одно Spring Boot приложение с несколькими доменными модулями. Модули имеют собственные ответственности и публичные контракты, но деплоятся как один artifact.

Единый backend artifact выбран потому, что он:

- упрощает локальную разработку;
- сокращает количество инфраструктурных компонентов;
- уменьшает время на CI/CD;
- позволяет использовать транзакции PostgreSQL для связанных бизнес-операций;
- снижает риск распределенных ошибок в первом релизе.

Модульность остается обязательной. Единый деплой не означает свободный доступ любого модуля к чужим данным или внутренним классам.

## Constraints

- На MVP не использовать микросервисы.
- На MVP не использовать Kafka.
- Не создавать отдельные сервисы без доказанного требования по нагрузке, командам или релизному циклу.
- Не допускать циклических зависимостей между модулями.
- Не превращать `common` в хранилище доменной логики.

## Future Evolution

После MVP возможен переход к Event Driven Architecture. Кандидаты на выделение: `receipts`, `payments`, `notifications`. Решение о выделении должно приниматься по фактам: нагрузка, независимый жизненный цикл, требования к надежности, необходимость внешних потребителей событий.

Kafka может быть добавлена только после стабилизации доменных событий и появления независимых потребителей. До этого внутренние вызовы и простые доменные события внутри монолита достаточны.
