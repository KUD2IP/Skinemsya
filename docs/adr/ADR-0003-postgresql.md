# ADR-0003: PostgreSQL

## Status

Accepted

## Purpose

Зафиксировать PostgreSQL как основную базу данных backend `skinemsya`.

## Context

Домен проекта транзакционный: пользователи состоят в группах, группы содержат мероприятия, мероприятия связаны с чеками, выбранными позициями, долгами и платежными операциями. Ошибки целостности напрямую влияют на деньги между пользователями.

Для MVP нужна надежная реляционная база с понятной локальной разработкой, хорошей поддержкой миграций, индексов и транзакций.

## Responsibilities

- Зафиксировать PostgreSQL как единый источник транзакционных данных.
- Объяснить отказ от NoSQL как основной БД на MVP.
- Поддержать проектирование связей и ограничений целостности.
- Дать основу для Liquibase-миграций и Testcontainers.

## Non Responsibilities

- ADR не определяет ERD.
- ADR не создает таблицы или индексы.
- ADR не выбирает managed database provider.
- ADR не описывает backup policy в деталях.

## Design Decisions

PostgreSQL используется как основная база данных. Она подходит для связанных доменных данных и позволяет использовать транзакции, внешние ключи, уникальные ограничения, индексы и миграции.

На MVP используется единая database для всего backend-приложения. При этом владение таблицами должно быть модульным: каждый модуль отвечает за свои таблицы и не должен напрямую изменять таблицы другого модуля.

## Constraints

- Все критичные бизнес-данные должны храниться в PostgreSQL.
- Нельзя использовать файловое хранилище как источник истины для чеков, долгов или платежей.
- Нельзя добавлять отдельную основную БД для модуля на MVP без отдельного ADR.
- Доступ к таблицам должен соответствовать модульному владению.
- Изменения схемы должны проходить через Liquibase.

## Future Evolution

После MVP возможны read replicas, partitioning, отдельные базы для выделенных сервисов или outbox pattern для событийной интеграции. Эти решения должны приниматься после появления нагрузки или требований, а не заранее.
